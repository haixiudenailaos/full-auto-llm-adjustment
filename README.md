# 全自动大模型调整

一个面向 Codex / Agent 工作流的通用 skill，用于在陌生项目、陌生仓库或高不确定性环境中，让智能体先通过用户互动收集线索，再建立本地行为契约，然后以小步快跑、可证伪、可复盘的方式持续推进任务。

这个 skill 不绑定某个具体竞赛、某个具体模型，或者某一种工程栈。它的目标是给智能体一个稳定的开局方式，避免一进入新环境就凭经验乱猜、盲扫仓库、在错误路径上反复试错。

## 它解决什么问题

大模型在陌生环境里最常见的失败，不是“不会写代码”，而是：

- 没搞清楚用户真正要的结果就开始行动
- 没问清楚历史背景、已试失败路径和隐藏约束
- 没有优先顺着用户给的线索找信息，而是盲扫整个环境
- 还没形成根因判断，就直接开始大改
- 在一条失败路线里反复调参数、调措辞、调局部细节
- 没有把已经确认的约束沉淀到 `AGENT.md` 或 `CLAUDE.md`

这个 skill 的设计目标，就是把这些失败模式前置拦住。

## 核心思路

这个 skill 采用的是一种“契约驱动 + 证据驱动 + 迭代驱动”的工作方式。

完整顺序不是“一上来先写文档”，而是：

1. 先和用户互动，尽可能多收集高价值线索
2. 根据这些线索去查本地文件、日志、脚本、历史记录、配置和外部信息
3. 再把已确认的约束写入 `AGENT.md` 或 `CLAUDE.md`
4. 然后按固定闭环继续推进：总结现状 -> 判断本质 -> 检查是否进入死路 -> 定向研究 -> 快速冒烟 -> 记录复盘

它的关键不是“写契约”本身，而是**在写契约前先校准环境认知**。

## 适用场景

这个 skill 适合以下类型的任务：

- 进入一个你之前没见过的项目仓库
- 接手一段文档不全、上下文残缺的工程任务
- 帮用户继续一个已经做过多轮尝试但方向混乱的项目
- 需要边探索边建立稳定工作规范的 agent 任务
- 需要通过多轮试验推进结果，而不是一次性直接写完的工作
- 需要在后续多次运行中复用同一套本地行为约束

不太适合的场景：

- 用户只要一个一次性、非常小的简单改动
- 项目环境非常清晰，已有明确 runbook，无需建立额外契约
- 用户只问纯概念问题，不涉及环境探索和执行推进

## 方法论拆解

### 1. 用户线索优先

skill 的第一原则不是“先扫代码”，而是“先从用户那里拿到最影响决策的线索”。

优先拿到的信息包括：

- 当前最重要的目标是什么
- 哪些事情已经试过并失败了
- 哪些文件、脚本、日志、命令最关键
- 真正的验收标准是什么
- 有没有隐藏约束、已有规则、既定偏好

这样做的原因很简单：在陌生环境里，用户通常知道“哪里最痛”，只是没有系统化表达。先把这些线索问出来，后续搜索和判断的成本会低很多。

### 2. 线索驱动取证

拿到用户线索之后，不应该盲扫环境，而应该顺着强线索先找证据：

- 指定目录
- 指定脚本
- 失败日志
- 历史实验记录
- 既有规范文件
- 配置、CI、依赖和运行入口

skill 强调“把模糊叙述变成可验证事实”，而不是靠模型自身经验补全。

### 3. 建立本地契约

当第一轮线索和证据已经足够时，就创建或更新 `AGENT.md` / `CLAUDE.md`。

这个契约文件不是 README，不是对人类写的说明书，而是给未来 agent run 用的本地行为约束层。它通常包含：

- `Goal`
- `Hard Constraints`
- `Source Priority`
- `Working Rules`
- `Validation`
- `Logging`

它的作用是让后续运行不必重复踩同样的坑。

### 4. Hermes 风格迭代

skill 采用一个固定闭环来避免无序试错：

1. `收集用户线索`
2. `总结现状`
3. `透过现象看本质`
4. `牛角尖检测`
5. `定向研究`
6. `快速冒烟`
7. `记录并复盘`

这套流程本质上要求 agent：

- 先分清事实和解释
- 先给出窄根因，再想方案
- 检查当前路线是不是已经走进死路
- 外部研究必须对准当前瓶颈，不允许泛搜
- 优先做最小可证伪实验，而不是直接大改
- 所有新认知都应该回写到日志和契约里

## 目录结构

当前仓库结构如下：

```text
full-auto-llm-adjustment/
├── SKILL.md
└── agents/
    └── openai.yaml
```

仓库根目录的 `README.md` 用于人类阅读；真正被 skill 系统使用的是：

- `full-auto-llm-adjustment/SKILL.md`
- `full-auto-llm-adjustment/agents/openai.yaml`

## 文件说明

### `full-auto-llm-adjustment/SKILL.md`

这是 skill 的核心定义文件，包含：

- skill 的触发描述
- 适用场景
- 行为原则
- 分阶段执行流程
- 迭代闭环
- 契约文件的更新原则

### `full-auto-llm-adjustment/agents/openai.yaml`

这是 UI / 触发层的元数据文件，包含：

- 展示名 `全自动大模型调整`
- 简短描述
- 默认 prompt

它决定这个 skill 在界面上如何展示，以及触发时默认向 agent 强调什么。

## 安装方式

如果你想把这个 skill 放进本地 Codex skills 目录，最直接的方式是把仓库中的 `full-auto-llm-adjustment/` 目录复制到你的 skills 根目录下。

典型目标路径类似：

```text
$CODEX_HOME/skills/full-auto-llm-adjustment
```

或者：

```text
~/.codex/skills/full-auto-llm-adjustment
```

安装后，确保目录中至少包含：

- `SKILL.md`
- `agents/openai.yaml`

## 使用方式

### 触发这个 skill

你可以通过类似下面的方式触发：

```text
$full-auto-llm-adjustment
```

或者在任务描述中明确表达这类意图：

- 进入一个陌生项目，先帮我摸清环境
- 先别急着改代码，先和我确认线索，再建立工作约束
- 这个仓库很乱，帮我边探索边沉淀 AGENT.md
- 根据我给你的线索继续推进，不要盲扫全部内容

### 推荐的使用心智

这个 skill 最适合配合下面这种协作方式使用：

- 用户先提供目标和若干关键线索
- agent 通过追问补足最关键的未知项
- agent 按线索去取证
- agent 写出第一版 `AGENT.md` / `CLAUDE.md`
- 然后开始进入闭环推进和复盘

## 这个 skill 对 agent 的要求

它并不是单纯让 agent “更主动”，而是要求 agent 更克制。

它要求 agent：

- 不要在信息不足时强行规划全局
- 不要把猜测当成环境规则
- 不要对同一路线做无意义微调
- 不要只汇报想法而不执行
- 不要只做局部代理验证而忽视真实目标

同时，它也要求 agent：

- 主动问关键问题
- 主动归纳事实
- 主动建立局部契约
- 主动设计最小实验
- 主动留下可复用记录

## 典型产物

使用这个 skill 后，理想情况下会留下这些产物：

- 一份 `AGENT.md` 或 `CLAUDE.md`
- 一份当前瓶颈摘要
- 一组经过筛选的下一步实验或验证动作
- 一份可复用的失败/成功记录

这意味着它不是只完成某一次对话，而是在为后续多轮智能体协作建立地基。

## 与普通“项目分析”提示词的区别

普通提示词往往只会说：

- 先分析项目
- 再提出方案
- 然后执行

这个 skill 更严格，它把“如何避免分析错方向”也内建进去了，尤其强调：

- 用户线索优先
- 证据优先于经验
- 契约文件作为长期对齐层
- 根因分类优先于方案发散
- 牛角尖检测优先于继续微调

也就是说，它不是普通的分析提示词，而是一套可持续运行的陌生环境推进协议。

## 注意事项

- 这个仓库当前主要包含 skill 本体，没有附带模板资产、脚本工具或额外参考资料。
- 如果后续要继续增强，可以再添加：
  - `AGENT.md` 模板
  - 迭代日志模板
  - 不同环境下的示例用法
  - 更细分的 reference 文档
- 如果你把它用于已有成熟流程的项目，建议让它优先复用现有 runbook，而不是重新发明一套规则。

## 仓库定位

这个仓库的定位不是“大模型调参工具”，而是一个**帮助大模型在陌生环境里自动进入正确工作状态的 skill**。

如果你要一句更准的概括，它更像：

> 一个面向陌生环境的、契约先行但用户线索优先的智能体迭代推进器。
